# Claude の外側に Context Graph を — ハーネスとして載せると速く、正確に、トークンを抑える

## この記事の要点

Claude Code や Cowork、[Claude Managed Agents](https://www.anthropic.com/engineering/managed-agents) など、業務文脈でもよく話題になります。プラットフォーム側ではエージェントの**ハーネス**（モデル外で、何を渡すか・誰に見せるかを束ねる）やコンテキスト設計が進んでいます。

**LLM 側のハーネス**（推論ループ・ツール配線）だけでなく、**企業データ側のハーネス**（メール・チャット・議事録・社内ドキュメントと CRM・各業務 DB などを**権限付きで束ねる層**）も要ります。業務データの出所や監査は、LLM 前提だけでは最適化しにくい。**Context Graph（文脈グラフ）** を **LLM の外側**に置く、という話から始めます。

そこでまず狙えるのは、**業務上は速く・正確に・トークンを抑える**ことです。話題はモデル比較に寄りがちですが、主題は乗り換えではなく、**同じ Claude でも渡す情報の質と量**をどう設計するかです。

**Context Graph** は事実・関係に加え状態や判断の経緯を **LLM の外**でつなぎ、「**いま渡すべき材料**」だけを選ぶ層です。ナレッジグラフや形式レイヤと同様、**LLM 本体から切り離して外側に置く**考え方です（[RAG を超える知識統合](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)、[LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture)）。市場・投資側がどう語っているかは、[Foundation Capital](https://foundationcapital.com/context-graphs-ais-trillion-dollar-opportunity) にまとまっています（英語）。

**Context Graph** の定義、**速さ・正確さ・トークン**、試すときの記録、実装例（DevRev）と動画、**自社に当てはめる目安**、という流れです。

> **より良く使う**＝モデル差し替えではなく、**LLM 側＋企業データ側（Context Graph）で質・量・権限**を整え、**速く・正確に・トークンを抑える**こと。

---

## コンテキストエンジニアリングとハーネス

**コンテキストエンジニアリング**は、**LLM に何を・どれだけ渡すか**を設計する仕事です（プロンプト、チャンク、ツール連鎖、権限別の見せ方、要約まで）。**ハーネス**はモデル外で見せる範囲や検証をルール化する層です。推論は LLM、再現性や監査はハーネスに寄せる、という分け方は [LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) と同じです。

**モデル内部で起きている推論そのもの**はブラックボックスに近く、業務の説明責任や監査の粒度ではそのまま検証しづらい、という側面があります。一方、**データ側のハーネス**に置いた参照・権限判断・モデルへ渡す直前の材料は、ログやスキーマとして**説明し、監査に耐えさせやすい**です。業務データは**モデルの外**（DB・グラフなど）に置き、**どこを書き換えの対象にするかもそこにそろえておく**と、**誤更新を元の状態に戻すロールバック**もやりやすくなります（モデル重みや推論内部の巻き戻しとは別の話です）。

[Anthropic のエンジニアリング記事](https://www.anthropic.com/engineering/managed-agents) の harness は主にループ・ツール・サンドボックス寄りです。**Context Graph** は **企業データ側のハーネス**として **LLM の外**に置き、データを束ねて渡す材料を選ぶ層です。参照の置き場は**モデル本体の外**です。過去記事のナレッジグラフ／形式レイヤと同じ切り口で、プラットフォーム harness と両立します。

プラットフォームに推論配線を寄せるのは合理的ですが、企業のデータ・権限はベンダ前提では最適化されにくく、「**モデル側のハーネスに全部**」ではデータ側に無理が出がちです。**企業データ側のハーネス**はプロンプト形式やツール API と**密結合**させず、**疎結合**に。**データの保存先を特定のモデルやベンダに縛らない**ことも、あわせて大事です。

いちばん短く言うと、コンテキストエンジニアリングは「**渡す中身**」、ハーネスは「**誰に何を渡していいかの枠**」。枠がないと**正しさ・速さ・トークン**がブレます。**Context Graph** は事実・関係・状態・イベントを一つの層に載せ、その両方の土台になります。外側の層が弱いと **速さ・正確さ・トークン**の伸びは**そこで止まりやすい**です。

## LLM の上限は、「コンテキスト」で決まる

LLM は与えられた**トークン列**の範囲で次を予測します。**どの事実・関係を渡すか**、**どの権限で何を見せるか**がずれると、推論が正しくても答えがずれます。

企業データは **構造化**（CRM・各業務 DB・レポートなど）と **非構造化**（メール・チャット・議事録・社内ドキュメント・会話ログ）に分かれ、**権限**もつきまといます。長いプロンプトに流し込むほどノイズとコストが増えるので、**コンテキストエンジニアリング**と**ハーネス**はそろえておきたいところです。

---

## Context Graph とは何か（定義）

**ナレッジグラフ**が「何が真か」「何とつながるか」の**意味構造**なら、**Context Graph** はその上に、エージェントや業務が参照すべき**動的な文脈**（「**いま・なぜ・誰が・どの経緯で**」）まで載せたレイヤとして、説明に使われることが多いです。

静的なエンティティと関係だけでは検索・推薦には強い一方、業務ではイベント・状態・**判断の経緯**（**Decision Trace**：誰がいつ、何を根拠にどう決めたかの痕跡）・監査の出典が分散するとエージェントが不安定になります。Context Graph は、これらを意味構造と一体で扱いたいというニーズに合わせて語られることが増えています。

本ブログでは **GraphRAG**（検索補強）と、意味構造そのものを扱う**ナレッジグラフ**を区別します（[RAG を超える知識統合：ナレッジグラフの役割](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)）。**Context Graph** は後者の延長で、**運用と監査に耐える文脈**まで含めたイメージです。

## なぜ速く、正確で、トークンを抑えられるのか

- **正確さ** … 事実・関係を**グラフ上のつながり**として渡せると取り違えが減り、**権限**に沿った見せ方にしやすい。
- **速さ** … MCP（Model Context Protocol）等で順に呼ぶと往復が積み上がる（[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph)）。データがつながった**記憶層**に問い合わせを集約できると、待ちの要因を減らしやすい。
- **トークン** … 入力は**事実・短い抜粋・ID**に寄せるほど、**コストと上限**に効く。ツール定義や中間結果の膨張は MCP 周辺でも論点になります（同上）。

### 試すときの記録（3 観点）

PoC でも本番でも、ログで「変える前／後」を見るのが確実です。

1. **速さ** … ツール・API 呼び出し回数、レイテンシ（P95 など）、レート制限への当たり。
2. **トークン** … 1 質問あたりの入力長（ツール説明・中間結果を含む）。
3. **正しさと権限** … 業務 ID・数値・参照箇所など**根拠**、閲覧権限の混入がないか。

経営向けには、トークン単価と人的手間の変化を同じ期間で並べると話しやすいです。PoC では「速さ・正確さ・トークンがどれもラインに届かないなら、データ統合や参照設計の前提から見直す」といった**打ち切り条件**を先に決めておくと、投資の迷いが減ります。

## LLM の外側に Context Graph を置く例

**Context Graph** を **LLM と切り離した外側**に置き、**渡す情報の質・量・権限**を整える、という整理です。実装は **ベンダ・クラウド・内製**のいずれもあり得ます。

以下では **DevRev** の Computer／Computer Memory を一例にします。顧客・案件・会話・製品情報などを権限付きで束ね、Claude へ渡す文脈を組み立てる、という位置づけです。一般化すると、**構造化／非構造化の同一参照層**、**権限付きの見せ方**、**推論はモデル・根拠や状態は外側のグラフ**、という線になります。[LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) の「形式レイヤで LLM を支える」ともつながります。思想を深く知りたいときの手がかりは [特許から読む DevRev の思想](https://zenn.dev/knowledge_graph/articles/devrev-philosophy-from-patents) です。

## 実例: 同じ Claude でも、LLM の外側に Context Graph を載せるとどう見えるか

製品名は出しますが、論点は自前・他社にも当てはまります。いちばん効くのはモデル名より**企業データ側のハーネス**です。[Claude Code](https://www.anthropic.com/claude-code) が進んでも、**事実・権限・履歴の渡し方**が固まらなければ、再現性やコストは伸びにくい、というのが体感に近いです。

### 観点別に何が変わりうるか（Claude を共通の前提に）

| 観点 | Claude のみ | Context Graph を外に載せる（例: Computer Memory） |
| ---- | ----------- | ---------------------------------------------------- |
| 根拠 | プロンプト断片やツール結果の取り違え、権限外の混入 | 業務レコードの ID・数値・該当箇所など、検証しやすい形で根拠を渡しやすい |
| 速度 | MCP 等で接続先を順に呼ぶと往復と待ちが積み上がる（[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph)） | 統合された記憶層へ問い合わせを集約し、レイテンシの要因を減らしやすい |
| 入力規模 | ツール定義・中間結果・長文貼り付けで入力が膨らみやすい | 必要な分だけに寄せやすい（計測はログで） |

#### 動画の内容（左画面と右画面）

同じ質問を **2 つのシステム**に投げ、**応答のプロセス**と**出てきた答えの質**を比べています。動画の流れに沿って整理します。左のツール接続（MCP／Skills の繋ぎ方など）は**この収録向けの一例**です。

**1. 質問の実行と思考プロセス**

ユーザーは「最近の **Intercom** のチケットから見える、顧客の苦情の**共通テーマは何か**」という質問を出します（英語プロンプトの意訳）。

- **左（Claude）** … **MCP（Model Context Protocol）** や **Skills**（Anthropic のスキル拡張）を通じて**各接続先**から情報を拾い、思考（thinking time）を**一つずつ積み上げる**プロセスが表示されます。
- **右（Computer）** … 背後でデータ同士の**関連**を素早く特定し、左より**短い思考時間**で回答生成に入っています。

**2. アウトプットの構造と具体性**

次の左右の差は、**あくまで本デモで見えた範囲の一例**です（表現や具体度は編集・環境で変わり得ます）。

生成結果を比べると、情報の**密度**と**形式**に差がはっきり出ます。

- **エグゼクティブ・サマリー（要約力）** … 右は **Executive Summary** と **Key Takeaways** がそろっていて、優先アクションが見える化されています。左は**リストの羅列**にとどまりやすい、という差です。
- **具体情報の抽出** … 左は「ポイントの付与がされていない」「Wi-Fi の不安定さ」など**抽象的なグルーピング**に寄りやすい一方、右は「**ゴールド／プラチナ会員に本来不要な手荷物料金が請求されている**」といった、**実際のチケットに踏み込んだ具体**を示しています。

**3. データ処理の効率性（Context Graph）**

この差の裏には、DevRev 側の **Context Graph** に相当する層があります。社内データを**あらかじめ構造化して保持**しておくので、LLM には「**いま読むべき情報**」だけをピンポイントで渡しやすくなります。

動画から拾える**3 つの利点**は次のとおりです（**録画時点・環境**で変わります）。

- **速度** … 検索・参照の範囲が絞られ、**回答までの体感**が速くなりやすい。
- **正確性** … データの関連が定義されているほど、**事実に根ざした具体的な**答えに寄りやすい。
- **リソース（トークン）** … 不要なデータを大量に読ませず、**処理の無駄（トークン消費）**を抑えやすい。

**4. 記事の論点との対応（見るときの目安）**

**1〜3** は、この記事の「速さ・正確さ・トークン」と対応しています。画面に入力長やトークンが出ていれば、「試すときの記録」と照らしてみてください。出ていなければ、待ち時間と根拠の見え方を見るとよいです。

## 自社に当てはめるとき（目安）

ツール連鎖（MCP 等）が複雑になるほど、先に決めたいのはデータ側です。**どのレコードを参照の起点にするか（ID・関係の一貫性）**と、**誰が何を見られるか（権限の境界）**。そのうえで、グラフやメタデータを使って「LLM に渡す量」を短く保てるかを見ます。

ナレッジグラフや業務 DB がある場合は、**渡す抜粋と監査ログの責務**が分かれているかも確認のポイントです。PoC では「試すときの記録」の 3 観点で**変更前後を同じ条件で比較**すると、続けるか止めるかの材料になります。

## まとめ

差がつきやすいのは **LLM の外側**です。**Context Graph** を **LLM と切り離して**置くと、同じ Claude でも**速く・正確に・トークンを抑えやすい**設計にしやすくなります。実装のしかたはさまざまで、小さく始めるなら **OSS でも試せます**。効果は環境と計測次第で、三つとも常に良くなるわけではありません。

- **整理**: **LLM 本体から切り離し、Context Graph を外側に置く**。意味構造に加え**動的な文脈**まで載せると、検索＋チャンク中心との違いが話しやすいです。
- **監査・復旧**: **モデル内部の推論**は業務監査の粒度では追いにくい一方、**データ側のハーネス**（参照・権限・渡す材料）はログやスキーマで説明しやすいです。業務データを**モデル外**に置き、更新単位をはっきりさせていれば、**誤変更を元に戻すロールバック**も現実的になりやすいです。
- **確かめ方**: **速さ・トークン・正しさと権限**をログで比較。[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph) に論点があります。
- **実例**: [Effortless Mumbai 2026（DevRev 公式）](https://www.youtube.com/watch?v=VViOXizcZx0) と [Claude と DevRev Computer の比較動画](https://youtu.be/WBmN4Cj2FLM) 。

形式レイヤの全体像は [LLM依存度を下げる業務AIアーキテクチャ設計](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) がわかりやすいです。
