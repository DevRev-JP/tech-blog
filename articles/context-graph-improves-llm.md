# Claude の外側にコンテキストグラフを置くと、速く・正確に・トークンを抑えられる

:::message
**前提知識**: この記事はナレッジグラフの基礎と RAG の限界を前提にしています。初めての方は先に「[ナレッジグラフ入門](https://zenn.dev/knowledge_graph/articles/knowledge-graph-intro)」と「[RAG を超える知識統合](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)」を読むとスムーズです。
:::

## この記事の要点

**LLM に渡す情報を外側で整理すると、同じモデルでも速く・正確に・安くなります。** その「整理する層」を本記事では **コンテキストグラフ** と呼びます。

LLM（大規模言語モデル）は、渡された情報の範囲内で回答を生成します。つまり「何を渡すか」の設計で回答の質が決まります。社内のメール・チャット・議事録・CRM などを、必要な分だけ・権限に沿って・根拠つきで渡せれば、モデルを差し替えなくても改善できます。

この記事では以下を順に説明します:

1. **用語と背景** — コンテキストエンジニアリング、ハーネス、コンテキストグラフの定義
2. **なぜ効くのか** — 速さ・正確さ・コストが改善する仕組み
3. **試し方** — PoC で何を計測すべきか（簡易コスト試算つき）
4. **実例** — DevRev Computer での比較動画と解説
5. **自社への当てはめ方** — チェックリスト

> より良く使う＝モデル差し替えではなく、LLM の外側で「質・量・権限」を整えて、速さ・正確さ・コストを改善すること。

## 用語の整理

本記事で繰り返し使う用語を先に定義します。

| 用語 | 一文定義 |
|------|----------|
| **トークン** | LLM が処理するテキストの最小単位。入力トークンが多いほど処理時間と費用が増える |
| **コンテキストエンジニアリング** | LLM に何を・どれだけ・どの順で渡すかを設計する仕事 |
| **ハーネス（harness）** | LLM の外側で「見せる範囲」「権限」「検証」をルール化する層。推論は LLM、再現性や監査はハーネスに寄せる |
| **MCP（Model Context Protocol）** | LLM がツールや外部データへ接続するための標準プロトコル |
| **コンテキストグラフ** | 事実・関係に加え、状態・判断の経緯まで LLM の外でつなぎ、「いま渡すべき材料」だけを選ぶ層 |

---

## コンテキストエンジニアリングとハーネス

コンテキストエンジニアリングは「LLM に何を・どれだけ渡すか」を設計する仕事です（プロンプト、チャンク、ツール連鎖、権限別の見せ方、要約まで）。ハーネスはモデル外で「見せる範囲」と「検証」をルール化する層です。推論は LLM、再現性や監査はハーネスに寄せる、という分け方になります。

モデル内部の推論はブラックボックスに近く、業務の説明責任や監査の粒度では検証しづらい側面があります。一方、データ側のハーネスに置いた参照・権限判断・「モデルへ渡す直前の材料」は、ログやスキーマで説明しやすい。業務データをモデル外（DB・グラフなど）に置き、更新対象もそこにそろえておくと、誤更新のロールバックもやりやすくなります（推論そのものを巻き戻す話ではありません）。

[Anthropic のエンジニアリング記事](https://www.anthropic.com/engineering/managed-agents) のハーネスは主にループ・ツール・サンドボックス寄りです。**コンテキストグラフ** は **企業データ側のハーネス**として **LLM の外**に置き、データを束ねて渡す材料を選ぶ層です。参照の置き場は**モデル本体の外**です。過去記事の[ナレッジグラフ](https://zenn.dev/knowledge_graph/articles/knowledge-graph-intro)／[形式レイヤ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture)と同じ切り口で、プラットフォームハーネスと両立します。

プラットフォームに推論配線を寄せるのは合理的ですが、企業のデータ・権限はベンダ前提では最適化されにくく、「モデル側のハーネスに全部」ではデータ側に無理が出がちです。企業データ側のハーネスはプロンプト形式やツール API と密結合させず疎結合にし、データの保存先を特定のモデルやベンダに縛らないことも大事です。

短く言うと、コンテキストエンジニアリングは「渡す中身」、ハーネスは「誰に何を渡していいかの枠」。枠がないと正しさ・速さ・トークンがブレます。コンテキストグラフは事実・関係・状態・イベントを一つの層に載せ、両方の土台になります。

---

## LLM の上限は、「コンテキスト」で決まる

LLM は与えられたトークン列の範囲で次を予測します。どの事実・関係を渡すか、どの権限で何を見せるかがずれると、推論が正しくても答えがずれます。

企業データは構造化（CRM・各業務 DB・レポートなど）と非構造化（メール・チャット・議事録・社内ドキュメント・会話ログ）に分かれ、権限もつきまといます。長いプロンプトに流し込むほどノイズとコストが増えるので、コンテキストエンジニアリングとハーネスはそろえておきたいところです。

## コンテキストグラフとは何か（定義）

[ナレッジグラフ](https://zenn.dev/knowledge_graph/articles/knowledge-graph-intro)が「何が真か」「何とつながるか」の意味構造なら、コンテキストグラフはその上に、エージェントや業務が参照すべき動的な文脈（いま・なぜ・誰が・どの経緯で）まで載せたレイヤとして説明されることが多いです。

静的なエンティティと関係だけでは検索・推薦には強い一方、業務ではイベント・状態・判断の経緯（`Decision Trace`：誰がいつ、何を根拠にどう決めたか）・監査の出典が分散するとエージェントが不安定になります。コンテキストグラフは、これらを意味構造と一体で扱いたいというニーズに合わせて語られることが増えています。

本ブログでは `GraphRAG`（検索補強）と、意味構造そのものを扱うナレッジグラフを区別します（[詳細はこちら](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)）。コンテキストグラフは後者の延長で、運用と監査に耐える文脈まで含めたイメージです。

## なぜ速く、正確で、トークンを抑えられるのか

- **正確さ**: 事実・関係を「グラフ上のつながり」として渡せると取り違えが減り、権限に沿った見せ方にしやすい。
- **速さ**: [MCP 等で接続先を順に呼ぶと往復が積み上がりやすい](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph)。データがつながった記憶層に問い合わせを集約できると、待ちの要因を減らしやすいです。
- **トークン（コスト）**: 入力は事実・短い抜粋・ID に寄せるほど、コストと上限に効きます。ツール定義や中間結果の膨張も、このあたりで効いてきます。

### 試すときの記録（3 観点）

PoC でも本番でも、ログで「変える前／後」を見るのが確実です。

1. **速さ** … ツール・API 呼び出し回数、レイテンシ（P95 など）、レート制限への当たり。
2. **トークン** … 1 質問あたりの入力長（ツール説明・中間結果を含む）。
3. **正しさと権限** … 業務 ID・数値・参照箇所など**根拠**、閲覧権限の混入がないか。

#### 簡易コスト試算の例

たとえば Claude Sonnet で社内 Q&A を処理する場合:

| | コンテキストグラフ なし | コンテキストグラフ あり | 差分 |
|---|---|---|---|
| 1質問あたり入力トークン | 約 8,000 | 約 2,500 | −69% |
| 月間質問数 | 10,000 | 10,000 | — |
| 月間入力トークン | 8,000万 | 2,500万 | −5,500万 |
| 月間入力コスト概算 | 約 $240 | 約 $75 | **−$165/月** |

※ 数値は仮定です。実際は業務・データ量で変わります。PoC では自社の平均入力長を計測し、同じ形式で比較してください。

経営向けには、トークン単価と人的手間の変化を同じ期間で並べると話しやすいです。PoC では「速さ・正確さ・トークンがどれもラインに届かないなら、データ統合や参照設計の前提から見直す」といった**打ち切り条件**を先に決めておくと、投資の迷いが減ります。

## LLM の外側にコンテキストグラフを置く例

コンテキストグラフを LLM と切り離した外側に置き、渡す情報の質・量・権限を整える、という整理です。実装はベンダ・クラウド・内製のいずれもあり得ます。

以下では DevRev の Computer／Computer Memory を一例にします。顧客・案件・会話・製品情報などを権限付きで束ね、Claude へ渡す文脈を組み立てる、という位置づけです。一般化すると「構造化/非構造化の同一参照層」「権限付きの見せ方」「推論はモデル、根拠や状態は外側のグラフ」という線になります。関連記事（[形式レイヤ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture)や [DevRev の思想](https://zenn.dev/knowledge_graph/articles/devrev-philosophy-from-patents)）も参考になります。

## 自社に当てはめるとき（目安）

ツール連鎖（MCP 等）が複雑になるほど、先に決めたいのはデータ側です。どのレコードを参照の起点にするか（ID・関係の一貫性）と、誰が何を見られるか（権限の境界）。そのうえで、グラフやメタデータを使って「LLM に渡す量」を短く保てるかを見ます。

最小チェックリスト（PoC でも本番でも共通）:

- 参照の起点となる ID と関係の一貫性があるか
- 権限境界（見せてよい/だめ）が機械的に判定できるか
- 根拠（参照レコードや該当箇所）がログに残るか
- LLM に渡す抜粋が短く保てるか（上限/コストを意識）
- 速さ・トークン・正しさ/権限を同条件で前後比較できるか

ナレッジグラフや業務 DB がある場合は、渡す抜粋と監査ログの責務が分かれているかも確認のポイントです。PoC では「試すときの記録」の 3 観点で変更前後を同じ条件で比較すると、続けるか止めるかの材料になります。

## まとめ

差がつきやすいのは LLM の外側です。コンテキストグラフを LLM と切り離して置くと、同じ Claude でも速さ・正確さ・トークンを改善しやすい設計に寄せられます。実装のしかたはさまざまで、小さく始めるなら OSS でも試せます。効果は環境と計測次第で、三つとも常に良くなるわけではありません。

- 整理: LLM 本体から切り離し、コンテキストグラフを外側に置く。意味構造に加え動的な文脈まで載せると、検索＋チャンク中心との差が話しやすいです。
- 監査・復旧: モデル内部の推論は追いにくい一方、データ側のハーネス（参照・権限・渡す材料）はログ/スキーマで説明しやすい。業務データをモデル外に置き、更新単位が明確ならロールバックもしやすいです。
- 確かめ方: 速さ・トークン・正しさ/権限をログで比較。
- 実例: Effortless Mumbai 2026（DevRev 公式）と比較動画。

形式レイヤの全体像は[こちらの記事](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture)で説明しています。
