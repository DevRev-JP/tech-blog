# Claude の外側に Context Graph を — ハーネスとして載せると速く、正確に、トークンを抑える

## 本稿の趣旨

Claude Code や Cowork、[Claude Managed Agents](https://www.anthropic.com/engineering/managed-agents) など、業務文脈でもよく話題になります。プラットフォーム側ではエージェントの**ハーネス**（モデル外で何を渡すか・誰に見せるかを支える足場）やコンテキスト設計が進んでいます。

**LLM 側のハーネス**（推論ループ・ツール配線）だけでなく、**企業データ側のハーネス**（メール・チャット・議事録・社内ドキュメントと CRM・各業務 DB などを**権限付きで束ねる層**）も要ります。業務データの出所や監査は LLM 前提では最適化されにくいので、**Context Graph（文脈グラフ）** を **LLM の外側**に置く、というのが本稿の入り口です。

その結果として狙えるのは、**業務上は速く・正確に・トークンを抑える**ことです。話題はモデル比較に寄りがちですが、本稿の主題は乗り換えではなく、**同じ Claude でも渡す情報の質と量**の設計です。

**Context Graph** は事実・関係に加え状態や判断の経緯を **LLM の外**でつなぎ、「**いま渡すべき材料**」だけを選ぶ層です。ナレッジグラフや形式レイヤと同様、**LLM 本体から切り離して外側に置く**立場です（[RAG を超える知識統合](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)、[LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture)）。投資・プロダクト側の語としての背景は [Foundation Capital](https://foundationcapital.com/context-graphs-ais-trillion-dollar-opportunity) 参照。

本論は **Context Graph** の定義、**速さ・正確さ・トークン**、試すときの記録、実装例（DevRev）と動画、**自社に当てはめる目安**、の順です。

> **より良く使う**＝モデル差し替えではなく、**LLM 側＋企業データ側（Context Graph）で質・量・権限**を整え、**速く・正確に・トークンを抑える**こと。

---

## コンテキストエンジニアリングとハーネス

**コンテキストエンジニアリング**は、**LLM に何を・どれだけ渡すか**を設計する仕事です（プロンプト、チャンク、ツール連鎖、権限別の見せ方、要約まで）。**ハーネス**はモデル外で見せる範囲や検証をルール化する層で、推論は LLM、再現性や監査はハーネスに寄せる、という分け方は [LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) に沿います。

**モデル内部で起きている推論そのもの**はブラックボックスに近く、業務の説明責任や監査の粒度ではそのまま検証しづらい、という前提があります。一方、**データ側のハーネス**に置いた参照・権限判断・モデルへ渡す直前の材料は、ログやスキーマとして**説明し、監査に耐えさせやすい**です。運用の**正本**が外側にあれば、**誤って変更されたデータを元の状態に戻すロールバック**もしやすくなります（モデル重みや推論内部の巻き戻しとは別の話です）。

[Anthropic のエンジニアリング記事](https://www.anthropic.com/engineering/managed-agents) の harness は主にループ・ツール・サンドボックス寄りです。**本稿の Context Graph** は **企業データ側のハーネス**として **LLM の外**に置き、データを束ねて渡す材料を選ぶ層です（参照点は**モデル本体の外側**。過去記事のナレッジグラフ／形式レイヤと同じ切り口）。プラットフォーム harness と両立します。

プラットフォームに推論配線を寄せるのは合理的ですが、企業のデータ・権限はベンダ前提で最適化されておらず、「**モデル側のハーネスに全部**」ではデータ側への無理が出がちです。**企業データ側のハーネス**をプロンプト形式やツール API と**密結合**させない（**疎結合**。正本は**モデル非依存**で持つ）ことが重要です。

短く言うと、コンテキストエンジニアリングは「**渡す中身**」、ハーネスは「**誰に何を渡していいかの枠**」。枠がないと**正しさ・速さ・トークン**がブレます。**Context Graph** は事実・関係・状態・イベントを一つの層に載せ、その両方の土台になります。外側の層が弱いと **速さ・正確さ・トークン効率**は頭打ちになりやすいです。

## LLM の上限は、「コンテキスト」で決まる

LLM は与えられた**トークン列**の範囲で次を予測します。**どの事実・関係を渡すか**、**どの権限で何を見せるか**がずれると、推論が正しくても答えがずれます。

企業データは **構造化**（CRM・各業務 DB・レポートなど）と **非構造化**（メール・チャット・議事録・社内ドキュメント・会話ログ）に分かれ、**権限**もつきまといます。長いプロンプトに流し込むほどノイズとコストが増えるので、**コンテキストエンジニアリング**と**ハーネス**を揃えるのが前提です。

---

## Context Graph とは何か（本稿での定義）

**ナレッジグラフ**が「何が真か」「何とつながるか」の**意味構造**なら、**Context Graph** はその上に、エージェントや業務が参照すべき**動的な文脈**（「**いま・なぜ・誰が・どの経緯で**」）まで載せたレイヤとして語られることが多い、という整理です。

静的なエンティティと関係だけでは検索・推薦には強い一方、業務ではイベント・状態・**判断の経緯**（**Decision Trace**：誰がいつ、何を根拠にどう決めたかの痕跡）・監査の出典が分散するとエージェントが不安定になります。Context Graph はそれらを意味構造と一体で扱いたい、というニーズの表れだと本稿では捉えます。

本ブログでは **GraphRAG**（検索補強）と、意味構造そのものを扱う**ナレッジグラフ**を区別します（[RAG を超える知識統合：ナレッジグラフの役割](https://zenn.dev/knowledge_graph/articles/beyond-rag-knowledge-graph)）。**Context Graph** は後者の延長で、**運用と監査に耐える文脈**まで含めるイメージです。

## なぜ速く、正確で、トークンを抑えられるのか

- **正確さ** … 事実・関係を**グラフ上のつながり**として渡せると取り違えが減り、**権限**に沿った見せ方にしやすい。
- **速さ** … MCP（Model Context Protocol）等で順に呼ぶと往復が積み上がる（[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph)）。データがつながった**記憶層**に問い合わせを集約できると、待ちの要因を減らしやすい。
- **トークン** … 入力は**事実・短い抜粋・ID**に寄せるほど、**コストと上限**に効く。ツール定義や中間結果の膨張は MCP 周辺でも論点になります（同上）。

### 試すときの記録（3 観点）

PoC でも本番でも、ログで「変える前／後」を見るのが確実です。

1. **速さ** … ツール・API 呼び出し回数、レイテンシ（P95 など）、レート制限への当たり。
2. **トークン** … 1 質問あたりの入力長（ツール説明・中間結果を含む）。
3. **正しさと権限** … 業務 ID・数値・参照箇所など**根拠**、閲覧権限の混入がないか。

経営向けには、トークン単価と人的手間の変化を同じ期間で見ると説明しやすいです。PoC では「速さ・正確さ・トークンのいずれも許容ラインに届かないなら、データ統合や参照設計の前提を見直す」といった**打ち切り条件**を先に置いておくと、投資判断の迷いを減らせます。

## LLM の外側に Context Graph を置く例

**Context Graph** を **LLM と切り離した外側**に置き、**渡す情報の質・量・権限**を整える、という話です。実装は **ベンダ・クラウド・内製**のいずれもあり得ます。

以下では **筆者の勤務先（DevRev）の Computer／Computer Memory** を一例にします（顧客・案件・会話・製品情報などを権限付きで束ね、Claude へ渡す文脈を組み立てる位置づけ）。一般化すると、**構造化／非構造化の同一参照層**、**権限付きの見せ方**、**推論はモデル・根拠や状態は外側のグラフ**、の線です。[LLM 依存度を下げる業務 AI アーキテクチャ](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) の「形式レイヤで LLM を支える」とも整合します。思想の詳細は [特許から読む DevRev の思想](https://zenn.dev/knowledge_graph/articles/devrev-philosophy-from-patents) 参照。

## 実例: 同じ Claude でも、LLM の外側に Context Graph を載せるとどう見えるか

製品名は出しますが、論点は自前・他社にも当てはまります。変えるのはモデル名より**企業データ側の足場**です。[Claude Code](https://www.anthropic.com/claude-code) 等が進んでも、**事実・権限・履歴の渡し方**が固まらなければ再現性やコストは頭打ちになりやすい、という立場です。

### 観点別に何が変わりうるか（Claude を共通の前提に）

| 観点         | Claude だけで企業データに触れるときの論点                                                                                                      | LLM の外側に Context Graph を載せたとき（一例として Computer Memory）に起きやすいこと |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 正確性・根拠 | プロンプト断片やツール結果の取り違え、権限外の混入                                                                                             | 業務レコードの ID・数値・該当箇所など、検証しやすい形で根拠を渡しやすい               |
| 速度         | MCP 等で接続先を順に呼ぶと往復と待ちが積み上がる（[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph)） | 統合された記憶層へ問い合わせを集約し、レイテンシの要因を減らしやすい                  |
| トークン     | ツール定義・中間結果・長文貼り付けで入力が膨らみやすい                                                                                         | 必要な分だけに寄せやすい（計測はログで）                                              |

#### 動画の内容（左画面と右画面）

同じ質問を **2 つの異なる AI システム**に投げ、**応答のプロセス**と**最終アウトプットの質**を比べています。動画内で起きている事実を、流れに沿って整理します。左側のツール接続（MCP／Skills の繋ぎ方など）は**この収録の構成**であり、自社の Claude 利用と**同一とは限りません**。

**1. 質問の実行と思考プロセス**

ユーザーは「最近の **Intercom** のチケットから見える、顧客の苦情の**共通テーマは何か**」という質問を出します（英語プロンプトの意訳）。

- **左（Claude）** … **MCP（Model Context Protocol）** や **Skills**（Anthropic のスキル拡張）を通じて**各接続先**から情報を拾い、思考（thinking time）を**一つずつ積み上げる**プロセスが表示されます。
- **右（Computer）** … 背後でデータ同士の**関連**を素早く特定し、左より**短い思考時間**で回答生成に入っています。

**2. アウトプットの構造と具体性**

次の左右の差は、**あくまで本デモで見えた範囲の一例**です（表現や具体度は編集・環境で変わり得ます）。

生成結果を比べると、情報の**密度**と**形式**に差がはっきり出ます。

- **エグゼクティブ・サマリー（要約力）** … 右には、状況を一望できる **Executive Summary** と、優先すべきアクションを示す **Key Takeaways** が自動で構成されています。左は**リスト形式の記述**にとどまりやすい、という対比です。
- **具体情報の抽出** … 左は「ポイントの付与がされていない」「Wi-Fi の不安定さ」など**抽象的なグルーピング**に寄りやすい一方、右は「**ゴールド／プラチナ会員に本来不要な手荷物料金が請求されている**」といった、**実際のチケットに踏み込んだ具体**を示しています。

**3. データ処理の効率性（Context Graph）**

この精度の差を支えているのが、DevRev 側の **Context Graph** に相当する層です。社内に散らばるデータを**あらかじめ構造化して保持**しているため、LLM には「**いま読むべき最適な情報**」だけをピンポイントで渡しやすい、という見取り方ができます。

動画から読み取れる**3 つの利点**は次のとおりです（**録画時点・環境依存**で体感や数値は変わり得ます）。

- **速度** … 検索・参照の範囲が絞られ、**回答までの体感**が速くなりやすい。
- **正確性** … データの関連が定義されているほど、**事実に根ざした具体的な**答えに寄りやすい。
- **リソース（トークン）** … 不要なデータを大量に読ませず、**処理の無駄（トークン消費）**を抑えやすい。

**4. 本稿との対応（見るときの目安）**

上記 **1〜3** は、本稿の「速さ・正確さ・トークン」に対応します。画面に入力長やトークンが出ていれば「試すときの記録」と照らして読んでください。出ていない場合は、レイテンシと根拠の見え方を主に見てください。

## 自社に当てはめるとき（目安）

ツール連鎖（MCP 等）が複雑になっているほど、**参照の正本と権限境界**をデータ側で先に押さえたうえで、グラフやメタデータで「何を LLM に渡すか」を短く保てるかを検討する、という順序が取りやすいです。既存のナレッジグラフや業務 DB がある場合は、**渡す抜粋と監査ログの責務分離**ができているかを確認し、PoC では「試すときの記録」の 3 つの観点で**変更前後を同条件で比較**すると、続行・中止の判断材料になります。

## まとめ

差がつきやすいのは **LLM の外側**です。**Context Graph** を **LLM と切り離して**置くと、同じ Claude でも**速く・正確に・トークンを抑えやすい**方向に設計しやすくなります。実装は多様で **OSS 前提ではありません**。効果は環境と計測に依存し、三つとも常に改善するとは限りません。

- **整理**: **LLM 本体から切り離し、Context Graph を外側に置く**。意味構造に加え**動的な文脈**まで載せると、検索＋チャンク中心との違いが説明しやすい（過去記事と同じ線）。
- **監査・復旧**: **モデル内部の推論**は業務監査の粒度では検証しにくい一方、**データ側のハーネス**（参照・権限・渡す材料）はログやスキーマで説明しやすい。正本が外側にあれば、**誤変更データを元に戻すロールバック**も現実的になりやすい。
- **確かめ方**: **速さ・トークン・正しさと権限**のログ比較。[MCPの課題とナレッジグラフ](https://zenn.dev/knowledge_graph/articles/mcp-knowledge-graph) 参照。
- **実例**: [Effortless Mumbai 2026（DevRev 公式）](https://www.youtube.com/watch?v=VViOXizcZx0) と [比較動画（合成版）](https://youtu.be/WBmN4Cj2FLM) 。

形式レイヤ全般は [LLM依存度を下げる業務AIアーキテクチャ設計](https://zenn.dev/knowledge_graph/articles/llm-formal-layer-architecture) を参照してください。
